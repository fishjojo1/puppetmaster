# Milestone 002: Codex Launch And Hooks

## Objective

Turn the generic tmux supervisor into a Codex-aware supervisor. Puppetmaster must be able to launch managed Codex sessions with the correct flags, generated configuration, MCP placeholders, environment variables, and Codex hooks. This milestone proves that managed Codex sessions can report lifecycle events back to Puppetmaster through hooks.

## Reader

This milestone is for the engineer implementing Codex runtime support. After reading it, they should be able to launch a real managed Codex process inside tmux and prove that its `Stop` hook calls back into Puppetmaster.

## Scope

This milestone includes:

1. Codex executable discovery.
2. Codex version detection.
3. Codex flag validation.
4. Per-agent launch wrapper generation.
5. Per-agent or per-root Codex config generation.
6. Per-agent hook script generation.
7. `Stop` hook handling.
8. `SessionStart` hook handling if useful.
9. Hook event persistence.
10. Initial prompt construction for managed agents.

This milestone excludes:

1. Full MCP tool implementation.
2. Orchestrator event continuation logic.
3. Sophisticated idle detection.
4. Subagent completion semantics beyond hook reporting.

## Required Behavior

Puppetmaster must launch Codex with:

```text
--no-alt-screen
--dangerously-bypass-approvals-and-sandbox
--cd <cwd>
```

The launch must set:

```text
PUPPETMASTER_AGENT_ID
PUPPETMASTER_PARENT_AGENT_ID
PUPPETMASTER_ROOT_AGENT_ID
PUPPETMASTER_STATE_DIR
PUPPETMASTER_CONFIG_DIR
PUPPETMASTER_ROLE
```

Every managed Codex session must have a `Stop` hook that calls Puppetmaster and records a durable `agent.turn_stopped` event.

The hook path must be generated by Puppetmaster rather than requiring the user to hand-edit global Codex config.

## Generated Files

Each agent should have generated files like:

```text
.puppetmaster/agents/<agent-id>/
  initial-prompt.md
  launch.sh
  codex-config/
    config.toml
    hooks.json
    hooks/
      stop-hook
```

`launch.sh` must be executable and should avoid leaking implementation details into tmux command strings. It should read the prompt from `initial-prompt.md` or otherwise pass the prompt to Codex safely.

## Hook Semantics

### Subagent `Stop`

When a subagent turn stops, the hook should:

1. Read Codex hook JSON from stdin.
2. Call Puppetmaster CLI with the calling agent id.
3. Store normalized hook input.
4. Append `agent.turn_stopped`.
5. Mark the agent `idle` or `awaiting_input` unless it already has a terminal completion status.
6. Return success to Codex without continuation.

### Orchestrator `Stop`

The orchestrator Stop hook can initially record turn stops exactly like subagents. The event-draining continuation behavior is implemented in milestone 004.

## Codex Config

The generated Codex config must enable hooks unless explicitly disabled:

```toml
[features]
hooks = true
```

The hook config should use `hooks.json` or inline TOML. Prefer whichever is easier to generate and inspect. The generated representation must be readable by humans.

Example shape:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/absolute/path/to/stop-hook",
            "timeout": 30,
            "statusMessage": "Reporting Puppetmaster turn stop"
          }
        ]
      }
    ]
  }
}
```

## Deliverables

1. Codex runtime launch service.
2. Generated launch script.
3. Generated Codex config.
4. Generated Stop hook.
5. CLI command to create a managed Codex agent.
6. Hook event handler command.
7. Validation proving a real Codex Stop hook records an event.
